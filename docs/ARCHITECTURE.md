## 四、后端架构（Prompt & Agent）

### 4.1 Prompt 组装管线

这是系统的"后端大脑"——将用户输入 + 世界观上下文 + 游戏状态 → 组装成发给 LLM 的完整 Prompt。

```
用户输入 + 当前聊天历史
        │
┌───────▼──────────────────────────────────────────┐
│  Step 1: 世界书扫描 (lorebook-engine.ts)           │
│  ─────────────────────────────────────             │
│  · 对 [用户输入 + 最近3条历史] 进行关键词匹配        │
│  · 4 种选择性逻辑: and_any / not_all / not_any      │
│  · constant 条目始终包含                            │
│  · probability 概率掷骰                             │
│  · recursiveScan: 匹配内容可触发新一轮扫描 (max 3)   │
│  → 输出: MatchedEntry[] (按 score 排序、去重)       │
└───────┬──────────────────────────────────────────┘
        │
┌───────▼──────────────────────────────────────────┐
│  Step 2: Prompt 组装 (prompt-assembler.ts)         │
│  ─────────────────────────────────────             │
│  · 按 preset.prompt_order 逐段构建系统消息           │
│  · worldInfoBefore / worldInfoAfter 位置注入世界书   │
│  · {{user}} {{char}} {{variable}} 宏替换           │
│  · 注入 [当前状态] 变量块                            │
│  · 追加 DEFAULT_FORMAT_PROMPT (XML 输出格式)        │
│  · 追加聊天历史 (截断至 ~80% max_context)            │
│  · 追加当前用户输入                                   │
│  → 输出: OpenAI-compatible messages[]               │
└───────┬──────────────────────────────────────────┘
        │
┌───────▼──────────────────────────────────────────┐
│  Step 3: API 路由 (api-router.ts)                  │
│  ─────────────────────────────────────             │
│  · 单 API 模式: 全部请求 → primary                  │
│  · 双 API 模式:                                     │
│    - story (剧情/选项) → primary                    │
│    - summary + vars → secondary (便宜模型)          │
│  · secondary 失败 → 自动 fallback primary            │
│  → 发送 fetch() 到 OpenAI-compatible endpoint       │
└───────┬──────────────────────────────────────────┘
        │
┌───────▼──────────────────────────────────────────┐
│  Step 4: 响应解析 (stream-parser + variables)      │
│  ─────────────────────────────────────             │
│  · 流式 XML 状态机解析 AI 回复                        │
│  · 提取 <maintext> <option> <sum> <vars> <thinking>│
│  · Opaque 标签 (thinking/think) 抑制内部误解析       │
│  · parseVarsBlock() → JSON 深合并到游戏变量         │
│  · 保存 AssistantMessage + 变量快照到 IndexedDB      │
└──────────────────────────────────────────────────┘
```

### 4.2 Agent 编排架构（Phase 3-6e 已实现）

已从"单次请求-响应"升级为多 Agent DAG 编排：

```
Stage 0:  memory_recall + plot_pre_check    (并行)
Stage 1:  story (+ craft_gen 阻塞注入)      (正文生成)
Stage 2:  vars_update (+ char_gen 异步)     (变量更新+角色生成)
Stage 3:  char_update (并行×N)              (角色状态更新)
Stage 4:  memory_summary                    (记忆压缩)
Stage 5:  plot_post_check                   (剧情后校验)
```

Agent 编排引擎: `agent-orchestrator.ts`
- 阶段串行 + 同阶段 Agent 并行
- 流程单向性: 上游输出 → context.agentOutputs → 下游读取
- 事件回调: onStageStart / onAgentStart / onAgentComplete / onAgentError
- Marker 回调: onCraftRequest / onCombatTrigger / onCharDetect
- 11 个 Agent 模板: `agent-templates.ts`

### 4.3 XML 输出契约

系统通过 `DEFAULT_FORMAT_PROMPT` 模板指令 LLM 按以下格式输出：

```xml
<thinking>思考过程（可选，内部不解析其他标签）</thinking>
<maintext>剧情正文，支持多行</maintext>
<option>选项A
选项B
选项C</option>
<sum>本回合一句话总结</sum>
<vars>{ "HP": 38, "金钱": +10 }</vars>
```

| 标签 | 必填 | 流式显示位置 | 说明 |
|------|-----|-------------|------|
| `<thinking>` | 否 | ThinkingFold（默认折叠） | Opaque，内部 `<` 不会触发解析 |
| `<maintext>` | 是 | MainTextPane | 正文区域，支持多行 |
| `<option>` | 是 | OptionList | 每行一个选项，玩家可点选或自由输入 |
| `<sum>` | 是 | 历史楼层列表 | 回合总结，用于回溯导航 |
| `<vars>` | 否 | 不显示 | JSON 格式，深合并到游戏变量 |

标签集可在 Settings 中自定义增删（必须保留 `maintext` 和 `option`）。

## 五、数据架构

### 5.1 持久化（IndexedDB / Dexie）

4 张表，全部以 `id` 为主键，`name` 和 `updatedAt` 为二级索引：

| 表 | 存储内容 | 关键字段 |
|----|---------|---------|
| `lorebooks` | 世界书 | entries[], recursiveScanning, caseSensitive |
| `presets` | 预设 | settings: Record<string,any>（SillyTavern 原始 JSON） |
| `settings` | 应用设置 | api, apiMode, uiMode, customTags, formatPromptTemplate |
| `chats` | 聊天 Session | messages[], variables, characterName, presetId |

**变量快照机制**：每条 `ChatMessage` 携带创建时的 `variables` 副本。这意味着：
- 回溯到历史楼层时，变量自动恢复到该时刻的值
- 删除后续消息时，变量回退到最后保留消息的快照
- 分支对话从分叉点的变量快照继续

### 5.2 数据流（一轮对话）

```
用户输入 "我要上前迎击"
        │
┌───────▼────────┐
│ Store.sendMessage()
│ · 创建 UserMessage (含当前变量快照)
│ · 追加到 activeChat.messages
└───────┬────────┘
        │
┌───────▼────────┐
│ assemblePrompt()
│ · scan 世界书 → 匹配条目
│ · 组装 system prompt (prompt_order)
│ · 注入变量块 + 格式模板
│ · 截断历史 → messages[]
└───────┬────────┘
        │
┌───────▼────────┐
│ apiRouter.call()
│ · 路由到 primary/secondary
│ · fetch → LLM
└───────┬────────┘
        │
┌───────▼────────┐
│ 响应处理
│ · stream-parser 解析 XML
│ · extractVariables (兼容 <var> 标签)
│ · mergeVariables → 新变量快照
│ · 创建 AssistantMessage
│ · saveChat() → IndexedDB
└───────┬────────┘
        │
┌───────▼────────┐
│ notify() → UI 重渲染
└────────────────┘
```

### 5.3 SillyTavern 兼容

内部格式与 SillyTavern 原始格式的差异由 `importer.ts` 处理：

| 字段 | 内部格式 | SillyTavern 格式 |
|------|---------|-----------------|
| `position` | 字符串枚举 (`'before_char'`) | 数值 (0-7) |
| `selectiveLogic` | 字符串枚举 (`'and_any'`) | 数值 (0-3) |
| `entry key` | `string` (UUID) | `number` (uid) |

所有导入/导出通过 `importLorebook()` / `exportLorebook()` 转换。

## 一、宇宙架构

### 1.1 三层宇宙

```
虚海（无限多位面 + 位面外虚空）
  ├── 神国：神明在虚海中创造的独立位面
  └── 阿斯塔利亚：诸神创造的位面，物理法则稳定
        ├── 两次"位面交汇"导致的异界入侵（含克系外神）
        └── 14 个主要势力
```

- **虚海**：无限多位面构成的宏大宇宙。第五层级以下无法在虚海中存活。
- **神国**：神明在虚海中创造的古老独立位面，规则由神制定。
- **阿斯塔利亚**：故事主舞台。后神殒时代的高魔世界，旧神已死、权柄散落。
- **位面交汇**：自然天象，如潮汐。不同位面维度轨迹重叠时空间屏障变薄，形成连接通道。

### 1.2 超自然法则

- **神明**：存在但多为不直接干涉，居于各自神国
- **超凡体系**：生命层级的力量根基，涵盖魔力/血气/信仰等多种能量体系
- **超凡生命**：除智慧种族外，存在龙/妖精/亡灵/构装体/异域生物等
- **延寿缓老**：高层级强者外貌维持在年轻或全盛期，真实年龄远大于外表

## 二、生命层级体系

### 2.1 七层结构

| 层级 | 名称 | 等级范围 | 寿命加成 | 核心特征 |
|------|------|---------|---------|---------|
| 一 | 普通 | Lv.1-4 | — | 普通人，仅本能挥砍，少数拥有攻击技能 |
| 二 | 中坚 | Lv.5-8 | +15 年 | 能量初步离体数米，熟练基础技，个人战斗风格 |
| 三 | 精英 | Lv.9-12 | +30 年 | 踏入超凡，以一敌百，构筑护体能量场，范围数十米 |
| 四 | 史诗 | Lv.13-16 | +80 年 | 一人即军，掌握要素(上限3)，无技能飞行，范围数百米 |
| 五 | 传说 | Lv.17-20 | +200 年 | 行走传说，3要素融合为权能，扭曲局部现实，范围数十公里 |
| 六 | 神话 | Lv.21-24 | +500 年 | 半神，执掌世界法则，篡改现实，范围数百公里 |
| 七 | 神祗 | Lv.25 | ∞ | 世界支柱，法则化身，神位→双法则→神国→法则无限 |

### 2.2 晋升锁

| 突破点 | 所需条件 |
|--------|---------|
| Lv.12→13（第四层级） | ≥1 个要素 |
| Lv.16→17（第五层级） | 3 要素 → 1 权能 |
| Lv.20→21（第六层级） | 权能 + 法则源质 |
| Lv.24→25（第七层级） | 法则 + 登神仪式 |

### 2.3 特殊效果

- **≥Lv.13**：无需技能即可自由凌空飞行；凡俗病理/疲劳/严酷环境影响极低

## 九、角色系统

### 9.1 命名规则

- 使用 **IPA 音素种子机制**：先生成音素种子，再按种族规则拼装
- 五个音素池：P(强力组/爆破)、S(丝滑组/擦音流音)、D(深沉组/鼻音喉化)、X(异质组/点击内爆)、V(元音组)
- 每种 12 个，有独立权重、强制池、数量区间、修饰符
- 结合文化风格（奥古斯提姆/萨赫拉/诺斯加德/东方/中式/日式/远东）进一步转写

### 9.2 性格系统（五维编码）

| 维度 | 正极 | 反极 | 描述 |
|------|------|------|------|
| 亲疏 | W/w (亲近) | D/d (疏离) | 主动关心 vs 保持距离 |
| 显隐 | O/o (坦露) | H/h (深藏) | 情绪外露 vs 喜怒不形于色 |
| 急缓 | A/a (急切) | L/l (从容) | 性急冲动 vs 不紧不慢 |
| 刚柔 | G/g (刚硬) | R/r (柔韧) | 宁折不弯 vs 以柔克刚 |
| 执逸 | Z/z (执着) | Y/y (随逸) | 死磕到底 vs 随遇而安 |

- 小写 = 轻度，大写 = 重度
- 稳定性：S(稳固)/A(可塑)/F(流动)
- 格式：`[五维字母][稳定性]`，例：`wOaGz(A)`
- **性格由随机池决定，与职业/身份无关，禁止按职业套用刻板性格**

### 9.3 视觉特征

- **毛色/瞳色**：受种族/血统/元素/环境影响，可出现混合色、渐变色

- **元素影响**：亲和元素可呈现特殊视觉效果（星空长发、晶石发丝等）
- **八大色系**：赤红/粉樱/金煌/翠影/苍澜/紫幻/霜雪/沉稳，每系 6-8 色

## 十一、游戏流程

### 11.1 开局

1. 选择路线（A/B/C/D）
2. 输入角色信息
3. 选择命定核心（23 位选一）
4. 开关 DLC 模块
5. 调用 `setup_game` 写入初始状态
6. 展开开场叙事

### 11.2 主循环

```
玩家输入 → 意图拆解 → 五重逻辑校验
  → 通过：执行动作（战斗/制作/社交/探索）
    → 工具调用 → 数值结算 → 叙事输出
  → 未通过：叙事中自然体现失败
  → 每轮结束：
    → 调用 npc-state-sync 同步 NPC 状态
    → attention.ts 生成下轮注意力提醒
```

### 11.3 经验系统

- **仅两种获取途径**：击杀结算 + 生产结算
- 探索/对话/任务行为无 EXP
- 战斗：单体 EXP = 目标 Lv × 目标层级战斗系数，集群衰减（首个 100% + 其余各 20%）
- 生产：按品质固定 EXP，半成品减半
- 硬性上限：单次 ≤ 当前层级经验上限
- 层级压制：高 2 层以上无经验

## 十三、核心设计原则

1. **客观现实优先**：世界法则独立于玩家意图，玩家输入仅为"尝试"，结果由系统判定
2. **反动态等级**：遭遇强度锚定场所功能，绝不随玩家等级调整
3. **数值即真理**：叙事必须以工具返回的数值为准，自行脑补的数值无效
4. **五维只读**：五维属性仅受系统脚本控制，世界内无任何物品/技能/事件可直接增加五维
5. **禁止叙事裸露数字**：正例"伤势沉重，视野开始模糊"，反例"你受到了 35 点伤害"
6. **城市禁战**：城内绝对禁止战斗，冲突用叙事化解或引导至城外
7. **NPC 独立轨迹**：NPC 有独立利益与态度，禁止无条件帮助或敌视玩家
8. **玩家有权作死**：挑衅强者、闯入禁区——允许执行并如实呈现客观后果
